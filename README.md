# Agent Team Orchestrator (ATO)

> 本地多 AI 智能体协作平台——在任意工作目录中创建和管理由多个 AI CLI 工具组成的智能体团队，通过协调者智能体自动分解和分派任务，实现多智能体协作完成长期项目需求。

## 功能概览

| 功能 | 说明 |
|---|---|
| **多智能体团队** | 创建包含多个智能体的团队，每个智能体可独立绑定不同的 AI CLI 工具 |
| **协调者调度** | 指定一个协调者智能体，自动将用户任务分解为子任务并分派给执行智能体 |
| **串行/并行执行** | 协调者可指定子任务串行（按序）或并行（同时）执行 |
| **实时日志监控** | 左侧面板实时展示每个智能体的状态和完整输出日志 |
| **超时检测** | 自动检测长时间无响应的智能体，提供继续等待、重试或中止三种干预选项 |
| **历史任务回放** | 完整记录每次任务执行过程，支持只读方式查看历史 |
| **配置持久化** | Team 配置自动保存到工作目录，重启后自动恢复 |
| **错误恢复** | 智能体进程异常时可一键重启，不影响团队整体配置 |

## 前提条件

### 必须安装

- **Node.js 18 或更高版本**
  ```bash
  node --version  # 应输出 v18.x.x 或更高
  ```

### 至少安装以下 CLI 工具之一

| CLI 工具 | 检测命令 | 安装说明 |
|---|---|---|
| **Claude Code CLI**（推荐） | `claude --version` | 参见 [Claude Code 文档](https://docs.anthropic.com/claude-code) |
| Gemini CLI | `gemini --version` | 参见 Google AI Studio |
| Codex CLI | `codex --version` | 参见 OpenAI Codex CLI |
| GitHub Copilot CLI | `gh copilot --version` | 参见 GitHub CLI 扩展 |

> **注意**：未安装的 CLI 工具在创建智能体时会显示警告，且该智能体无法正常工作。Copilot CLI 主要面向命令建议场景，用于通用对话任务时效果可能有限。

## 安装与启动

### 安装步骤

```bash
# 1. 下载项目到本地
git clone <repo-url> ato
cd ato

# 2. 安装所有依赖（前端 + 后端一次性安装）
npm install

# 3. 启动开发模式
npm run dev

# 仅启动后端（可选）
npm run dev:backend

# 仅启动前端（可选）
npm run dev:frontend
```

启动成功后，在浏览器中访问：**http://localhost:5172**
> 前端端口已锁定为 `5172`（strictPort），端口被占用时会直接报错，不会自动切到 5173。

### 生产模式启动

```bash
# 构建前端静态文件
npm run build

# 启动服务器（同时服务前端和后端）
npm start
```

### 自定义端口

```bash
# 修改后端端口（默认 3000）
ATO_PORT=3001 npm start
```

> 若修改了后端端口，还需同步修改 `frontend/.env` 中的 `VITE_BACKEND_PORT` 值。

### Windows 一键启动/关闭/状态检查

项目根目录提供 `run.bat`，双击或命令行执行：

```bat
run.bat
```

菜单功能：
- 一键启动前后端（`npm run dev`）
- 单独启动后端 / 单独启动前端
- 一键关闭前后端 / 单独关闭后端 / 单独关闭前端（按端口停止）
- 状态检查（进程 + 端口监听）

说明：
- 前后端可一键同时启动，也可分开启动；默认端口固定：后端 `3000`，前端 `5172`。
- 前端就绪后才自动打开浏览器（5172）。
- 若启动失败，请查看新打开的 `ATO Dev` 终端窗口输出。

│  左侧面板（320px）   │         右侧主区域                   │
│                     │                                    │
│  ● 智能体列表        │  工作目录 / 团队配置                 │
│    ├ Coordinator    │                                    │
│    │  [工作中 ●]    │  任务输入区                         │
│    └ Developer      │  ┌─────────────────────────────┐   │

└─────────────────────┴────────────────────────────────────┘

- **左侧面板**：可点击 `◀` 按钮折叠，展开后点击单个智能体可查看其详细日志

  - 🟢 绿色：已连接后端服务
  - 🔴 红色：连接中断（自动重连中）
  - 🟡 黄色：正在连接

## 快速上手（5 分钟入门）

### 第一步：指定工作目录

在界面右上角的"工作目录"输入框中，输入你要在其中工作的项目目录的**绝对路径**，例如：
- Windows：`C:\Users\YourName\my-project`
- macOS/Linux：`/home/yourname/my-project`

点击"加载"。

- 若该目录已有 `.agent-team` 配置文件 → 自动加载已有团队
- 若该目录没有配置 → 提示创建首个团队

### 第二步：创建团队

点击"创建团队"按钮，填写配置：

1. **团队名称**：例如"开发团队"
2. **添加智能体**：点击"添加智能体"：
   - **名称**：例如"Coordinator"（协调者）
   - **角色描述**：描述该智能体的职责，例如"你是一个任务协调者，负责分解用户需求并分派给团队成员"
   - **CLI 工具**：从下拉框选择已安装的 CLI 工具
   - **设为协调者**：勾选此项（**每个团队必须有且只有一个协调者**）
3. **再添加一个执行智能体**：例如"Developer"，不勾选协调者

> 💡 提示：可以从预设角色模板快速填充角色描述（后端开发、前端开发、测试工程师、架构师）

4. 点击"保存"——配置自动写入工作目录下的 `.agent-team` 文件

### 第三步：提交任务

在右侧"任务描述"输入框中输入任何自然语言任务，例如：

```
检查当前目录下所有 Python 文件的语法，并生成一份简要报告
```

点击"执行任务"。

### 第四步：观察执行过程

- 左侧面板：智能体状态由"空闲"变为"工作中"（蓝色脉冲动画）
- 右侧消息流：实时显示智能体之间的消息传递过程
- 任务完成后：最终结论显示在消息流底部

### 第五步：查看历史

任务完成后，点击左侧"历史记录"，找到刚完成的任务，点击"查看"可以以只读方式回放整个执行过程。

## 功能详解

### 团队管理

#### 创建团队

每个工作目录只能创建一个团队。首次在未配置的目录加载时，界面会自动引导创建流程。

**智能体配置项说明**：

| 配置项 | 必填 | 说明 |
|---|---|---|
| 名称 | ✅ | 智能体的唯一标识名，分派时使用此名称路由消息 |
| 角色描述 | ✅ | 告诉 AI 这个智能体的职责是什么，影响其行为和输出风格 |
| CLI 工具 | ✅ | 绑定到哪个 AI CLI 工具（需已本地安装） |
| 设为协调者 | — | 整个团队的任务调度中枢，有且只能有一个 |

#### 编辑团队

在左侧面板顶部点击"编辑"按钮，可修改团队名称、增删智能体、修改角色描述。

> ⚠️ 注意：任务执行中不允许修改团队配置。

#### 删除/切换团队

删除 `.agent-team` 文件后重新加载工作目录，即可重新创建新团队。

---

### 提交与执行任务

#### 提交任务

1. 确认工作目录已加载且连接状态为绿色
2. 在任务输入框中输入自然语言描述（支持多行）
3. 点击"执行任务"

#### 执行流程说明

```
用户输入任务
    ↓
发送给协调者智能体
    ↓
协调者决定：
    ├── 直接回复 → 显示最终结果
    └── 分派子任务 → 发送给执行智能体
            ↓
        执行智能体处理各自子任务
            ↓
        结果汇总返回给协调者
            ↓
        协调者给出最终结论 → 显示最终结果
```

每一轮协调-执行过程都会实时显示在右侧消息流中。

#### 中止任务

任务执行中，点击"中止任务"按钮可立即终止所有正在运行的子进程并标记任务为已中止。

---

### 实时状态监控

#### 智能体状态

左侧面板中每个智能体卡片显示当前状态：

| 状态 | 颜色 | 含义 |
|---|---|---|
| 空闲 | 灰色 ○ | 等待任务，可正常工作 |
| 工作中 | 蓝色脉冲 ● | 正在执行子任务，CLI 进程运行中 |
| 错误 | 红色 ✕ | 上次调用失败，需要干预 |

#### 查看详细日志

点击任意智能体卡片，左侧面板展开显示该智能体的完整日志：

- **发送**（蓝色）：调度引擎向该智能体发送的消息
- **接收**（绿色）：该智能体返回的输出
- **系统**（灰色）：状态变化、进程事件等系统消息
- **错误**（红色）：错误信息和异常记录

日志采用虚拟滚动，即使有数百条也不影响性能。

---

### 超时检测与干预

ATO 每 15 秒检查一次所有工作中智能体的最后输出时间。若某个智能体超过 **120 秒**无任何输出，前端弹出超时提示。

**三种干预选项**：

| 选项 | 行为 |
|---|---|
| 继续等待（+120秒） | 延长超时阈值，继续等待该智能体输出 |
| 重试该子任务 | 终止当前进程，重新向该智能体发送相同子任务 |
| 中止整个任务 | 终止所有活跃进程，任务标记为已中止 |

**协调者可调整超时**：对于已知耗时较长的任务，协调者可在分派指令中包含 `<timeout seconds="300"/>` 来延长超时时长（最大 1800 秒）。

---

### 历史记录与回放

#### 查看历史列表

点击左侧面板的"历史记录"标签，显示当前工作目录下所有历史任务，包括：
- 任务描述（截断预览）
- 执行时间
- 任务状态（已完成/失败/已中止）

#### 回放历史任务

点击历史任务右侧的"查看"按钮，进入只读回放模式：
- 完整显示任务描述
- 按时间顺序展示所有智能体的消息交换记录
- 显示最终输出结论

> ⚠️ 历史记录为**只读**，不支持对历史任务进行编辑或重新执行。

#### 历史文件位置

历史记录保存于工作目录下的 `.agent-history/` 文件夹，文件名为 `<任务ID>.json`，可直接打开查看原始数据。

---

### 智能体错误恢复

当某个智能体的 CLI 调用失败（如 CLI 未安装、API Key 过期、进程崩溃）时：

1. 左侧面板该智能体状态变为红色"错误"
2. 点击该智能体卡片查看错误详情
3. 排查并解决问题后（如重新配置 API Key），点击"重启"按钮
4. 适配器实例重建，状态恢复为"空闲"，可继续参与后续任务

## 配置文件格式

### `.agent-team`（团队配置）

保存于工作目录根目录，JSON 格式，可用文本编辑器手动查看（不建议直接编辑）：

```json
{
  "version": "1.0",
  "teamName": "开发团队",
  "workdir": "/path/to/your/project",
  "agents": [
    {
      "id": "abc123",
      "name": "Coordinator",
      "role": "你是一个任务协调者，负责分解用户需求并分派给团队成员。",
      "cli": "claude-code",
      "isCoordinator": true,
      "systemPrompt": "（保存时自动生成，含分派协议说明）"
    },
    {
      "id": "def456",
      "name": "Developer",
      "role": "你是一个后端开发专家，负责代码实现任务。",
      "cli": "claude-code",
      "isCoordinator": false,
      "systemPrompt": "你是一个后端开发专家，负责代码实现任务。"
    }
  ],
  "coordinatorId": "abc123",
  "createdAt": "2026-01-01T00:00:00.000Z",
  "updatedAt": "2026-01-01T00:00:00.000Z"
}
```

**`cli` 字段可选值**：`claude-code`、`copilot`、`codex`、`gemini`

### `.agent-history/<task-id>.json`（历史记录）

```json
{
  "id": "task-uuid",
  "description": "检查当前目录下所有 Python 文件的语法",
  "status": "completed",
  "createdAt": "2026-01-01T10:00:00.000Z",
  "completedAt": "2026-01-01T10:03:45.000Z",
  "finalOutput": "共检查 12 个 Python 文件，发现 2 个语法错误：...",
  "logs": [
    {
      "ts": "2026-01-01T10:00:01.000Z",
      "agentId": "abc123",
      "agentName": "Coordinator",
      "type": "send",
      "content": "请帮我检查当前目录下所有 Python 文件的语法..."
    }
  ]
}
```

## 协调者分派协议（进阶）

协调者智能体在需要分派子任务时，会自动输出以下格式的 XML 指令块（已在保存团队时自动注入到协调者的系统提示词中）：

```xml
<ato-dispatch>
  <!-- 串行任务：等待上一个串行任务完成后再执行 -->
  <task agent="Developer" mode="serial" id="task-001">
    实现用户登录接口，返回 JWT token
  </task>

  <!-- 并行任务：与其他 parallel 任务同时执行 -->
  <task agent="Tester" mode="parallel" id="task-002">
    为登录功能编写单元测试
  </task>
  <task agent="Frontend" mode="parallel" id="task-003">
    实现登录表单 UI 组件
  </task>

  <!-- 可选：覆盖超时时长（秒） -->
  <timeout seconds="300" />
</ato-dispatch>
```

**你不需要手动编写这个格式**——协调者 AI 会自动遵循注入的格式说明。如果协调者不需要分派任务（直接能回答），它会直接输出文字回复，调度引擎会将其作为最终答案展示。

### 提升协调质量的技巧

1. **明确描述期望的分工**：在创建团队时，给协调者的角色描述中说明团队成员的专长
2. **任务粒度适当**：太宏观的任务（如"做一个完整的网站"）会让协调者难以分派；建议拆分为具体可执行的模块
3. **告知团队能力**：在任务描述中可以提示"请分别让前端工程师和后端工程师负责各自部分"

## 常见问题

**Q：应用启动后页面白屏或报错怎么办？**
A：检查 Node.js 版本是否 ≥18（`node --version`），并确认已运行 `npm install`。查看终端输出的错误信息定位问题。

**Q：加载工作目录后提示"配置文件读取失败"？**
A：确认输入的是绝对路径（如 `/home/user/project` 而不是 `~/project`），且该路径存在且有读写权限。

**Q：智能体一直处于"错误"状态？**
A：点击该智能体卡片查看错误详情。最常见原因：
- CLI 工具未安装或不在 PATH 中（运行 `claude --version` 等命令验证）
- API Key 未配置或已过期
- 工作目录权限问题

**Q：任务执行后没有分派给其他智能体，协调者直接给出了答案？**
A：这是正常行为——协调者判断该任务无需分派时会直接回复。如果期望触发分派，可在任务描述中明确要求，例如"请分别让 A 负责 X，让 B 负责 Y"。

**Q：如何在不同项目之间切换？**
A：在工作目录输入框中输入新的项目路径后点击"加载"即可，每个目录维护各自独立的 `.agent-team` 配置。

**Q：能同时运行多个工作目录的团队吗？**
A：当前版本每次只支持一个活跃工作目录。切换工作目录会停止当前任务。

**Q：`.agent-team` 文件可以手动编辑吗？**
A：文件是标准 JSON 格式，技术上可以编辑，但不建议直接修改 `systemPrompt` 字段，因为它由应用自动管理。建议通过界面配置。

**Q：历史记录占用太多磁盘空间怎么办？**
A：历史文件保存在工作目录的 `.agent-history/` 文件夹中，可以直接删除不需要的历史文件。

## 安全说明

- ATO 后端仅监听 `127.0.0.1`（localhost），不对外暴露任何网络端口，无法从其他设备访问
- 所有 CLI 子进程的工作目录强制限定为指定工作目录，不允许路径跨越
- ATO 本身不收集或上传任何数据；所有 AI 调用均由你本地安装的 CLI 工具直接发出
- 不包含用户认证，**请勿在公共或共享网络环境中使用**
